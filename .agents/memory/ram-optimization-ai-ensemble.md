---
name: RAM optimization for AI ensemble
description: Why AI engine model sizes/retrain cadence were reduced, and where the real memory ceiling is
---

The bot is also deployed on an external low-memory host (Bothost, Docker, GitHub-synced) separate from this Replit env, with a much tighter RAM ceiling (~256-512MB). That host was OOM-killing the gunicorn worker seconds after boot, during the initial `pretrain()` model-fitting phase.

**Why:** sklearn/xgboost/lightgbm ensembles hold all fitted estimators in memory permanently, and during (re)fit, old+new model objects plus training arrays can coexist, spiking well above steady-state RSS. `pretrain()` fits 6-7 models sequentially at startup — that spike is what triggers OOM on constrained hosts, not steady-state usage.

**How to apply:** if OOM/RAM issues resurface, first check whether it's a startup-time spike (during pretrain/refit) vs. steady-state growth. For spikes: shrink `n_estimators`/`max_iter`/`max_depth` per model, shrink `REPLAY_SIZE`, raise `RETRAIN_EVERY`, and call `gc.collect()` right after each `slot.fit()` (already done in `ai_engine.py`). Note the Python+numpy+pandas+sklearn+xgboost import baseline alone is ~200-250MB RSS even before any model exists — that floor can't be reduced without dropping a whole library (e.g. XGBoost or LightGBM), which is a bigger tradeoff (fewer models = quicker but slightly less accurate ensemble). If OOM persists after these tunings, the real fix is more container RAM on the external host, not further code shrinking.

**Follow-up fix (2026-07-06) — instant repeat-SIGKILL loop despite prior shrinking:** two more root causes were still active: (1) `gc.collect()` alone doesn't return freed arenas to the OS under glibc malloc — RSS can stay high even after GC; fixed by adding `ctypes.CDLL("libc.so.6").malloc_trim(0)` right after every `gc.collect()` (see `_release_memory()` in `ai_engine.py`), plus a periodic call from the trader's main tick loop. (2) the ensemble was still 6 heavy models (RF/ET/GB/HGB/XGB/MLP) even after shrinking their individual sizes — added a `LOW_MEMORY_MODE` env var (set in `Dockerfile` for Bothost only, unset on Replit) that drops to just 3 small models (RF/ET/GB, ~40-60 estimators) and a smaller `REPLAY_SIZE`/`RETRAIN_EVERY`. If a host's hard RAM cap is tight enough, per-model shrinking isn't sufficient — dropping whole models via a host-specific flag is the next lever, not just tuning existing ones.

**Follow-up fix #2 (2026-07-06, same day) — SIGKILL loop persisted even with LOW_MEMORY_MODE=1 already deployed:** the flag disabled *using* HGB/XGB/LGB models but `import xgboost`/`import lightgbm` still ran unconditionally at module load — importing those libraries alone costs tens of MB of RSS even if no model instance is ever created. Fixed by gating the imports themselves behind `if LOW_MEMORY_MODE: skip` in `ai_engine.py`, not just the model-construction branch. Also cut gunicorn `--threads` 8→4 and pretrain candle-fetch limit 300→150 under `LOW_MEMORY_MODE` (each thread's stack/locals and the feature-matrix size both scale RSS). **Lesson: an env-flag that only gates object *construction* is insufficient for RAM control — always check whether the *import itself* of an optional heavy library still runs unconditionally.** Also: this bot's Bothost deployment pulls from a separate GitHub repo (`github.com/Alexkkkkk/GRINCH-GRAM`) via Docker — Replit-side fixes only take effect there after a `git push` to that repo AND a manual redeploy on Bothost; the agent cannot trigger that redeploy itself.

**Resolution (2026-07-06):** user confirmed the Bothost container actually has 4 vCPU / 2GB RAM — far more than the ~256-512MB this whole optimization assumed. Repeated SIGKILL loops at that RAM level, even with the 3-model LOW_MEMORY_MODE ensemble, point away from a real OOM and toward Bothost enforcing a stricter per-container cgroup limit than the account-level plan advertises (plausible on a stated "beta" multi-tenant platform). Per user's explicit choice, `LOW_MEMORY_MODE` was set back to `0` in the `Dockerfile` (full 7-model ensemble, 8 gunicorn threads, 300-candle pretrain) to prioritize AI accuracy over the memory-defensive posture.

**Reverted again same day — full ensemble caused production outage:** within ~15 minutes of the full-ensemble deploy going live on Bothost, logs showed repeated `Handling signal: term` (platform-initiated SIGTERM, not gunicorn's own SIGKILL-on-OOM) roughly every 2-11 minutes, and the container eventually failed to restart at all (site returned the *proxy's* 404, meaning traffic never reached the app — a dead/unroutable container, not an app-level error). This is a different signature from the original SIGKILL-loop bug and points at Bothost's own platform monitor killing the container for exceeding a real resource quota tighter than the advertised 4vCPU/2GB. Reverted `LOW_MEMORY_MODE` back to `1` and gunicorn threads back to 4. **Lesson: on this specific Bothost plan, do not trust the advertised container specs — treat `LOW_MEMORY_MODE=1` (3-model ensemble) as the stable baseline for that host until Bothost support confirms the actual enforced memory/cgroup limit.** Full ensemble remains fine on Replit dev and any host with confirmed headroom.

**Hard floor established (2026-07-06):** user asked to push LOW_MEMORY_MODE down to ~120MB RSS. Measured: numpy+pandas+sklearn import alone costs ~200-250MB RSS before any model exists (confirmed via isolated test) — this is a hard floor of the library stack itself, not reducible without dropping pandas/numpy/sklearn entirely (i.e. rewriting the AI engine without ML libraries, a full rewrite with real accuracy loss). 120MB is not achievable while keeping the sklearn ensemble. Squeezed LOW_MEMORY_MODE further instead (realistic target ~180-220MB in full app context, verified ~159MB for the isolated model-training path alone): shrank the 3 LOW_MEMORY_MODE models further (RF 60→35, ET 50→30, GB 40→25 estimators), cut `REPLAY_SIZE` 300→200, added `CONFIRMED_CAP=250` to bound the previously-unbounded `_confirmed_X/_y/_w` trade-history buffer (both on append and on disk-restore — this buffer grows forever otherwise, one entry per closed trade, over months of uptime), and shrank `analytics_buffer.py` tick/trade ring buffers (400→200, 100→60) under `LOW_MEMORY_MODE`. **Lesson: always give a numeric memory-floor estimate for a given library stack before promising a memory target — Python+numpy+pandas+sklearn imports alone are ~200-250MB regardless of app code size.**

**Full-history-without-RAM-cost pattern (2026-07-06):** instead of trimming `CONFIRMED_CAP` at the cost of losing old training examples, added an append-only DB table (`bot_ai_examples`, no cap) written on every confirmed trade outcome; the in-RAM buffer stays capped for hot/live fitting, and a separate background thread (`Trader._start_deep_retrain_thread`, ~every 2 days) pulls a large window (2000) from the DB and does one full refit, then lets RAM return to the capped steady state. **Why:** sklearn needs data resident in RAM at fit time, so you can't avoid a temporary spike during any full refit — but that spike only needs to happen rarely (days apart), not persist. **How to apply:** for any "don't lose data under a RAM cap" problem, prefer DB-backed durable history + periodic bulk refit over either (a) uncapping the live RAM buffer or (b) permanently discarding old data.
